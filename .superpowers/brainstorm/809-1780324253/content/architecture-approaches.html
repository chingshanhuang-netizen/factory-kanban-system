架構方案選擇

請選擇最適合工廠看板系統的架構方案

A

單一 RCL 專案（Monolithic）

所有功能（地圖、看板、資料源、服務）全部放在同一個 Razor Class Library 專案中。

優點

  • 部署簡單，只需一個 DLL
  • 開發初期速度快
  • 專案結構單純

缺點

  • 專案龐大後難以維護
  • 各層耦合度高
  • 單元測試困難
B

分層多專案解決方案（推薦）

拆分為三個專案，嚴格對應 Controller-Service-View 架構：

專案結構
📦 TPS.Nexus.Kanban.Core
Models、Interfaces、Enums、共用合約
📦 TPS.Nexus.Kanban.Services
業務邏輯、資料源適配器（SQL/CSV/JSON/XML）
地圖解析器、版本管理、SignalR Hub
📦 TPS.Nexus.Kanban.Web (RCL)
Razor 元件、拖拉佈局、看板 Widget、頁面

優點

  • 清晰邊界，符合 CSV 架構
  • Service 層可獨立單元測試
  • 各層可獨立更新

缺點

  • 多個專案需管理
  • 初期設定較多
C

微插件架構（Micro-plugin）

核心看板模組 + 可插拔資料適配器 + 可插拔 Widget 類型，每種地圖格式、每種資料源各為獨立插件。

優點

  • 最大彈性與可擴展性
  • 可單獨更新某一資料源適配器
  • 未來新增格式無需改動核心

缺點

  • 架構複雜度高
  • 對目前需求過度設計
  • 開發週期較長
